# delilog-website プロジェクトガイド

delilog（軽貨物ドライバー向け業務記録アプリ）の公式サイトおよびブログ。Next.js App Router + Markdown でブログを運用。

---

## プロダクト事実（記事との整合性必須）

ブログ記事内で delilog の機能・仕様に言及する場合、以下の事実と矛盾しないこと。

### アプリの機能

| 機能 | 仕様 | 注意 |
|------|------|------|
| 点呼記録 | アルコール「使用有無」「酒気帯び有無」の2項目ボタン、所見欄（任意） | **数値入力欄は意図的に設けていない**（国交省モデル様式準拠） |
| 所見欄 | 初期値は空白 | **記載は任意**。異常時のみ入力推奨 |
| 稼働なし表示 | 記録のない日は PDF 出力時に「稼働なし」と自動表示 | 休業理由を入力する機能はない（法令上も不要） |
| 日常点検 | 15項目チェックリスト | **無料プラン** |
| 業務の記録（運行記録） | 開始・終了・休憩の日時と地点、走行距離、経過地点 | **プレミアム機能**（14日間無料トライアル） |
| 勤務時間管理 | 改善基準告示に基づく自動チェック | プレミアム機能 |
| 事故の記録 | 6項目の記録＋3年保存 | プレミアム機能 |
| PDF出力 | **国交省モデル様式に準拠** | 無料・プレミアム共通 |

### 配布

- iOS: https://apps.apple.com/jp/app/delilog/id6753698337
- Android: 近日公開（Google Play版は現時点で未リリース）
- 料金: 基本無料、プレミアムに14日間無料トライアルあり

### やってはいけない記述

- delilog に「アルコール数値入力欄がある」と書く
- delilog に「休業理由を記録する機能がある」と書く
- 点呼記録に「数値の記載が法令上必須」と書く
- 点呼所見欄の「毎日の具体記載が法令上必須」と書く

## 法令記述のルール

### 一次資料のみ採用

法令・規制に関する主張は、次のソースのみで検証する。二次情報（ブログ記事・民間解説サイト）は参考止まりとし、記事の根拠として引用しない。

**採用可能なソース**:

- e-Gov法令検索（laws.e-gov.go.jp）
- 国土交通省（www.mlit.go.jp）
- 警察庁（www.npa.go.jp）
- 国家公安委員会
- 全日本トラック協会（jta.or.jp）※業界団体の公式標準様式として
- 各都道府県トラック協会（公式様式・通達のみ）

**採用不可**:

- SmartDrive, 一番星倶楽部, パイ・アール, Bqey などの民間ブログ解説（参考のみ、直接引用不可）
- AI要約の二次情報
- 「一般に言われている」など出典不明の主張

### 法令改正日は絶対に確認

- 軽貨物への安全対策強化は**2025年4月1日施行**（経過措置なし）
- 点呼記録の保存期間は**1年**、事故の記録は**3年**
- 改善基準告示の数値はルール改定があり得るため、毎回 e-Gov で最新値を確認

## 記事の構造テンプレート

既存3記事が確立したパターン。新記事もこの型を踏襲。

### Frontmatter

```yaml
---
title: "軽貨物「XXX」完全ガイド｜N項目の書き方・〇〇・△△【2026年版】"
description: "2025年4月施行のXXX。〇〇、△△、□□まで、個人事業主ドライバー向けにやさしく解説します。"
date: "YYYY-MM-DD"
slug: "lowercase-hyphenated-slug"
ogImage: "/blog/<slug>/og.png"
tags:
  - 法令対応
  - 軽貨物
  - <追加タグ>
author: "delilog編集部"
---
```

### 文字数

- 目安 **7,000〜10,000字**（空白除外）
- 短すぎ: SEO・独自性で弱い
- 長すぎ: 読了率低下

### 章構成パターン

```
[OG画像]

(リード段落: 読者の疑問に共感する問いかけ)

> この記事でわかること（箇条書き4〜5点）

> 関連記事への誘導（シリーズ相互リンク）

---

## 1. なぜこの義務・ルールがあるか（背景）
## 2. 対象・期限・罰則・保存期間
## 3. 項目の全体像（表＋画像1枚）
## 4. 項目別の具体的な書き方（記入例付き）
## 5. つまずきポイント3選（画像1〜2枚）
## 6. 記録方法（紙/Excel/アプリ比較）
## 7. よくある質問（FAQ）7問程度
## 8. まとめ＋delilog CTA

---

## あわせて読みたい（シリーズ相互リンク）

## 参考資料（一次資料のURL列挙）
```

### Voice / Tone

- **丁寧語、だが親しみ**。「〜ください」ではなく「〜しましょう」「〜がおすすめです」
- **読者の疑問に先回り**。「〜と感じるかもしれませんが」「〜と思いがちですが」で共感
- **「結論: 〇〇」で明確に断言**。法令準拠の範囲で
- **絵文字・装飾文字は使わない**。テキストだけで読みやすくする
- **delilog への言及は控えめに**。§6と§8のCTA部分のみで自然に誘導

### シリーズ内部リンク

既存記事（各 slug）:

- `keikamotsu-anzentaisaku-2025` — 14項目の全体像
- `gyoumu-no-kiroku-kakikata` — 業務の記録6項目
- `tenko-kiroku-kakikata` — 点呼記録4項目

新記事を追加したら:
1. 新記事の冒頭「この記事でわかること」後に既存記事へのリンクを設置
2. 新記事の末尾「あわせて読みたい」で既存記事を引用
3. **既存記事の末尾「あわせて読みたい」を編集**し、新記事の「（近日公開）」→実リンクに更新

## デザインシステム（OG画像・記事内画像）

### カラーパレット（厳密）

| 用途 | カラーコード |
|------|-------------|
| クリーム背景 | `#F6EDD9` |
| プライマリオレンジ | `#EA5A1A` |
| ネイビーブルー | `#1E3A5F` |
| 薄オレンジ（ハイライト背景） | `#FCE8DD` |
| 白 | `#FFFFFF` |
| マットグレー（副次要素） | `#B8B0A0` |

### 画像仕様

- **OG画像**: 1200×630px、PNG、150KB以下目標
- **記事内画像**: 1200×675px（16:9）、PNG、1MB以下目標
- **配置**: `public/blog/<slug>/*.png`
- **alt属性**: 必ずSEOを意識した具体的な説明を設定

### OG画像テンプレート（Canvaで作成）

既存パターン（1本目〜3本目）を踏襲:

- 左上: delilog ロゴカード
- 中央: メインタイトル2行（1行目: `軽貨物ドライバーの`、2行目: `[テーマ] 書き方ガイド` などオレンジ＋ネイビー配色）
- 左下: バッジ「2026年版」「国交省 告示対応」+ サブタイトル
- 右下: 巨大オレンジカードに**項目数の数字**（過去: 14/6/4）＋「項目」縦書き

### 記事内インフォグラフィック

フラットデザイン。影・グラデーション不使用。角丸8〜12px。日本語 Noto Sans JP Bold ベース。必ず STYLE REFERENCE プリアンブルをプロンプトに含める。

## SEO / 検索対策

### ターゲットKW例

- `軽貨物 [トピック] 書き方`
- `[トピック] 個人事業主 義務`
- `[トピック] 記録方法`
- `黒ナンバー [トピック]`

### ページ内対策

- H1 には必ず主要KWを含める
- description（meta）にKWを自然に含める
- alt属性にKWを含める（ただし過剰最適化NG）
- 内部リンクは具体的なアンカーテキスト（記事タイトル）で

### Search Console

- 新記事公開後、手動でインデックス登録リクエスト
- 推奨URL: **UTM無しのcanonical URL**
- `https://delilog.app/blog/<slug>` を Search Console → URL検査

## ビルド・デプロイ

- Vercel 自動デプロイ（`main` push で即反映、2〜3分）

## Git コミット規則

- Conventional Commits 風: `feat: / fix: / docs: / refactor:`
- 新ブログ記事: `feat: N本目のブログ記事「XXX」を追加`
- コミット本体には**変更の要点**（新規追加ファイル・既存記事への相互リンク更新・法令解釈の根拠）を箇条書き
- `Co-Authored-By` は無効（global `~/.claude/settings.json` で disabled）

各エージェントの責務と連携は `.claude/commands/write-blog.md` を参照。
